Three-Phase Pipeline

Phase 1

Architecture Recon

Map the target Java application's attack surface before vulnerability analysis.

  • Framework detection (Spring, JAX-RS, Servlet, etc.)
  • HTTP entry point discovery
  • Data flow source identification
  • Trust boundary mapping
  • Endpoint auth classification
  • Database & external integration inventory
  • High-risk component flagging

Output: .github/sast-context.md

→
Phase 2

Vulnerability Analysis

Independent vulnerability skills run in parallel, each taint-tracking source to sink.

  • 31 vulnerability-class skills
  • Function tree + function review + taint trees
  • Attack chain construction
  • API abuse & business logic research
  • Each skill references Phase 1 context

Output: .github/sast-findings.jsonl

→
Phase 3

Reporting

Consolidate all findings, deduplicate, rank by severity, and generate reports.

  • Schema validation
  • Root-cause deduplication
  • Severity & exploitability ranking
  • 200-component structured report
  • Static HTML report generation

Skill: sast-report

Outputs: .github/sast-report.md, .github/sast-report.html

Severity Definitions

CRITICAL HIGH MEDIUM LOW INFO
SeverityCriteriaCVSS Band
CRITICALRemote, unauthenticated, exploitable with a single HTTP request. Data exfiltration, RCE, auth bypass at the application perimeter.9.0–10.0
HIGHAuthenticated exploit, or chained with one other issue. Lateral movement, privilege escalation, significant data exposure.7.0–8.9
MEDIUMRequires specific conditions (race window, admin context, non-default config). Limited blast radius.4.0–6.9
LOWDefense-in-depth gap, best-practice violation, no direct exploitability. Informational hardening items.0.1–3.9
INFOObserved pattern, not a vulnerability. Worth noting for architecture review.—

False Positive Suppression

  • PreparedStatement with correct parameterization
  • ObjectInputStream inside test harnesses with no network surface
  • Hardcoded credentials that are clearly test/mock values
  • Path operations on classpath-derived resources only
  • Reflected XSS in application/json endpoints

Java-Specific Always-On Rules

  • csrf().disable() → always flagged (MEDIUM minimum)
  • @Transactional bypassed via direct field injection → suspicious
  • @RequestMapping without auth annotation → potential unauthenticated endpoint
  • @SuppressWarnings("unchecked") on deserialization code → extra scrutiny
  • Lombok @Data on JPA entities with bidirectional relationships → LOW if Jackson present

Git Hooks

Pre-Commit Hook

Lightweight review before committing Java changes.

Triggers on: *.java, *.xml, *.properties, *.yml, *.yaml, pom.xml, build.gradle, build.gradle.kts

  1. Inspect only changed files and related callers/callees
  2. Run sast-analysis if entry points, security config, or deps changed
  3. Run relevant Phase 2 skills for changed surfaces (SQL→sqli, auth→auth/idor/csrf, XML→xxe, etc.)
  4. Report only actionable findings with changed-line evidence

Blocking: Only Critical or High confidence findings block the commit.

Pre-Push Hook

Broader review before pushing Java application changes.

  1. Refresh sast-context.md via sast-analysis
  2. Run all matching Phase 2 skills
  3. Run sast-dependency if manifests changed
  4. Consolidate with sast-report
  5. Highlight new Critical or High findings first

Blocking: Confirmed Critical and remotely-reachable High findings block the push.
